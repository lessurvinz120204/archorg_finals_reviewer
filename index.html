Master Memory Hierarchy, Memory Management & I/O Systems

Overview & Purpose

Memory Hierarchy is a system of organizing different types of memory based on speed, cost, and capacity. It minimizes access time by utilizing the principle of locality of references - the idea that data or nearby data is likely to be accessed again. The hierarchy ranges from ultra-fast CPU registers to slow secondary storage.

Levels of Memory Hierarchy (Fastest to Slowest)

Level 0: CPU Registers - <1KB, 0.25-0.5ns access time, managed by compiler Level 1: Cache Memory - <16MB, 0.5-25ns access time, SRAM-based Level 2: Main Memory (RAM) - <16GB, 80-250ns access time, DRAM Level 3: Magnetic Disk - >100GB, 50+ milliseconds, disk storage Level 4: Optical Disk - Large capacity, slower access Level 5: Magnetic Tape - Backup storage, slowest access

Key Characteristics

• Capacity: Increases from top to bottom (Registers < Cache < RAM < Disk) • Access Time: Increases from top to bottom (Registers fastest, Tape slowest) • Cost Per Bit: Decreases from top to bottom (Registers most expensive) • Performance: Faster memory near CPU improves system performance • Cost Efficiency: Balance between speed and storage capacity

Memory Types

Static RAM (SRAM): Fast, stores in flip-flops, used for cache, no refresh needed Dynamic RAM (DRAM): Larger capacity, requires refresh circuitry, used for main memory Secondary Storage: Hard drives (HDD), SSDs, optical disks - non-volatile

Address Binding

Logical Address: Generated by CPU during execution (virtual address) Physical Address: Actual location in memory Address Binding: Process of mapping logical to physical addresses Three Stages of Binding: 1. Compile-time: Done by compiler before loading 2. Load-time: Done by OS loader when program loads into memory 3. Execution-time (Dynamic): Done by MMU during program execution MMU (Memory Management Unit): Hardware that performs runtime address translation

Swapping & Dynamic Loading

Swapping: Process of temporarily moving data from main memory to disk (backing store) - Allows total physical space to exceed actual physical memory - Increases multiprogramming degree - Backing store must be large enough for all memory images Dynamic Loading: Load routines only when called, not at startup - Improves memory space utilization - Routines kept in relocatable format on disk - Linking loader called when routine is needed

Contiguous Memory Allocation

Divides memory into partitions - each process occupies one contiguous block Fixed Partitioning: Equal-sized partitions, simple but inflexible Dynamic Partitioning: Variable-sized partitions based on process needs Allocation Strategies: • First Fit: Allocate first free block large enough • Best Fit: Allocate smallest free block that fits (minimizes waste) • Worst Fit: Allocate largest free block (maximizes remaining space)

Fragmentation

Internal Fragmentation: Unused space within allocated block - Occurs when process smaller than allocated partition - Problem with fixed partitioning - Solution: Best-fit allocation External Fragmentation: Non-contiguous free blocks cannot satisfy requests - Occurs with dynamic partitioning - Solutions: Compaction, paging, segmentation - Compaction: Reorganize memory to combine fragmented blocks

Segmentation

Non-contiguous memory allocation using logical segments Compiler creates segments: code, global variables, heap, stacks, libraries Segment Address: Segment Table: Maps segment number to base address and limit Advantages: • No internal fragmentation • Smaller segment table than page table Disadvantages: • External fragmentation when processes loaded/unloaded • Complex to implement

Paging

Non-contiguous memory allocation using fixed-size pages Main memory divided into frames, secondary memory into pages Page Address: Page Table: Maps page number to frame number PTBR (Page Table Base Register): Points to page table location Page Offset: Doesn't need translation (same as frame offset) TLB (Translation Lookaside Buffer): Fast cache for page-to-frame mappings - TLB Hit: Mapping found, no page table access needed - TLB Miss: Access page table, load mapping into TLB Advantages: No external fragmentation, efficient swapping Disadvantages: Internal fragmentation possible, page table overhead, 2 memory accesses

Virtual Memory & Demand Paging

Virtual Memory: Illusion of unlimited memory by using disk as extension Allows processes larger than physical memory Demand Paging: Load pages only when referenced, not entire process Uses lazy swapper (Pager): Never swaps page unless needed Page Fault: Occurs when referenced page not in memory Steps: Check validity → Find free frame → Load page → Update tables → Restart instruction Advantages: Less I/O, less memory needed, faster response, more concurrent users

Page Replacement Algorithms

FIFO (First In First Out): • Simple, low overhead • Poor performance, suffers from Belady's Anomaly • More frames can increase page faults Optimal (OPT): • Replace page not used for longest time • Best performance but needs future knowledge • Complex to implement, impractical Least Recently Used (LRU): • Replace page not used in most time • Uses past knowledge, practical implementation • Better than FIFO, worse than OPT Thrashing: Condition where OS spends most time servicing page faults - CPU becomes idle - Increasing multiprogramming increases thrashing - Decreases utilization significantly

I/O Systems Overview

Definition: Bridge between computer and external environment Enables data exchange with peripherals and users Why I/O Matters: • Critical for system performance • Essential for user interaction • Enables communication with outside world Key Concepts: • Data Transfer Rate: Speed of data movement • Bandwidth: Maximum data transfer capacity • Buffering: Temporary storage during transfer • Polling: CPU repeatedly checks device status • Interrupts: Device signals CPU when ready

Types of I/O Devices

Character Devices: • Sequential data transfer (keyboard, mouse) • Data transferred one character at a time Block Devices: • Fixed-size block storage (hard drives, SSDs) • Data transferred in blocks Communication Devices: • Network interface cards (NICs) • Enable data exchange over networks Sensors & Actuators: • Embedded system components • Physical interaction with environment

I/O System Components

I/O Modules & Controllers: • Specialized hardware managing CPU-to-peripheral communication • Handle data buffering and protocol translation • Act as intermediary between CPU and devices System Buses: • Data Bus: Carries information between components • Address Bus: Specifies memory/device locations • Control Bus: Manages operations and timing CPU-Memory Bus: Connects CPU and main memory I/O Bus: Connects I/O controllers to system Bus Adapter: Bridges different bus types

Data Transfer Methods

Programmed I/O: • CPU directly controls all data transfers • Processor continuously polls device status • Simple implementation but CPU-intensive • Not suitable for high-speed transfers Interrupt-Driven I/O: • CPU interrupted only when device ready • Processor free for other tasks while waiting • More efficient than programmed I/O • Still involves CPU in data transfer Direct Memory Access (DMA): • Dedicated DMA controller handles transfers • CPU completely offloaded from transfer • Ideal for high-speed, bulk data transfers • Essential for disk drives and network cards • Improves system performance significantly

I/O Challenges

CPU-Device Synchronization: • Speed difference between fast CPU and slow peripherals • Requires handshaking protocols and wait states • Risk of data loss if not managed carefully Multi-Device Management: • Handling multiple devices simultaneously • Efficient arbitration schemes needed • Priority handling to prevent conflicts • Error detection mechanisms required Speed Mismatch Problems: • CPU much faster than I/O devices • Creates bottlenecks requiring buffering • Asynchronous communication protocols needed • Polling/interrupts/DMA handle this
